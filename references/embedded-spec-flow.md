# 内置 Spec-Flow 结构化开发工作流

本文件是 `spec-flow` skill 的自包含版本，供 WorkCopilot 在用户未安装独立 spec-flow skill 时直接使用。

## 核心规则

1. **一次一阶段**：只做当前阶段，禁止提前生成后续文档
2. **强制确认**：每阶段完成后必须停下等待用户说 "continue" / "继续" / "好" / "下一步"
3. **中文输出**：所有 .md 文件内容使用中文
4. **快速模式**：用户说 "fast mode" / "快速模式" 时可连续执行，但最后仍需确认

## 确认模板

每阶段完成后使用此格式：

```
📋 **[阶段名称] 完成**

已创建 `.spec-flow/active/{feature}/{file}.md`，包含：
- [关键内容摘要]

**请审阅**：
1. [审阅问题]？
2. [审阅问题]？

✅ 说 "继续" 进入下一阶段
✏️ 告诉我需要修改的内容
```

## Phase 1: 提案（Proposal）

**目标**：定义 WHY — 为什么要做这个

**产出文件**：`.spec-flow/active/{feature}/proposal.md`

**内容结构**：

```markdown
# {功能名称} 提案

## 背景
{变更的上下文和动机}

## 目标
- [ ] 目标 1（可量化）
- [ ] 目标 2

## 非目标
- {明确排除的内容 1}
- {明确排除的内容 2}

## 范围
### 包含
- ...
### 不包含
- ...

## 风险
| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| ... | ... | ... |

## 开放问题
- [ ] 问题 1
- [ ] 问题 2
```

**⏸️ 确认点**：背景是否准确？目标是否清晰？范围是否合理？

---

## Phase 2: 需求（Requirements）

**目标**：定义 WHAT — 系统应该做什么

**产出文件**：`.spec-flow/active/{feature}/requirements.md`

**使用 EARS 格式**：
- **普遍型**：`系统应当...`
- **事件驱动**：`当 [触发条件] 时，系统应当...`
- **状态驱动**：`在 [状态] 下，系统应当...`
- **异常行为**：`如果 [条件]，则系统不应当...`

**内容结构**：

```markdown
# {功能名称} 需求规格

## 功能需求

### FR-001: {需求标题}
**类型**：事件驱动
**描述**：当 [触发] 时，系统应当 [行为]
**优先级**：P0 / P1 / P2
**验收标准**：
- AC-001: Given [前置条件], When [操作], Then [预期结果]

### FR-002: ...

## 非功能需求

### NFR-001: 性能
- 页面加载时间 < 2s
- API 响应时间 P95 < 500ms

### NFR-002: 安全
- 所有 API 需要认证
- 敏感数据加密存储

### NFR-003: 可靠性
- 系统可用性 > 99.5%
```

**⏸️ 确认点**：功能需求是否覆盖所有用例？非功能需求是否充分？验收标准是否可测试？

## Phase 3: 设计（Design）

**目标**：定义 HOW — 如何实现

**产出文件**：`.spec-flow/active/{feature}/design.md`

**内容结构**：

```markdown
# {功能名称} 技术设计

## 架构概览
{Mermaid 架构图}

## 组件设计
{每个组件的职责和接口}

## API 设计
| 方法 | 路径 | 描述 | 请求体 | 响应体 |
|------|------|------|--------|--------|
| POST | /api/v1/... | ... | {...} | {...} |

## 数据模型
{Mermaid erDiagram}

## 错误处理
| 错误码 | HTTP 状态 | 描述 | 处理方式 |
|--------|----------|------|---------|
| ... | ... | ... | ... |

## 迁移计划
{数据/Schema 迁移步骤（如适用）}
```

**⏸️ 确认点**：架构是否满足所有需求？API 设计是否清晰？有无遗漏的边界情况？

---

## Phase 4: 任务拆解（Tasks）

**目标**：分解为可执行的步骤

**产出文件**：`.spec-flow/active/{feature}/tasks.md`

**任务拆解原则**：
- 每个任务在 1-2 个工具调用内可完成
- 包含复杂度评估：Low / Medium / High
- 列出涉及的文件
- 标明任务间依赖关系
- 按阶段分组：Setup → Implementation → Testing → Documentation

**内容结构**：

```markdown
# {功能名称} 任务清单

## 阶段 1: 基础搭建

### T-001: {任务标题}
- **复杂度**：Low
- **Agent**：backend-engineer
- **文件**：`src/...`
- **依赖**：无
- **状态**：⏳ 待开始

### T-002: ...
- **依赖**：T-001

## 阶段 2: 核心实现
...

## 阶段 3: 测试
...

## 阶段 4: 文档
...
```

**进度标记**：⏳ 待开始 → 🔄 进行中 → ✅ 完成 → ❌ 阻塞

**⏸️ 确认点**：任务粒度是否合适？依赖关系是否正确？准备开始实施吗？

---

## Phase 5: 实施（可选）

支持三种执行模式：

| 模式 | 触发词 | 行为 |
|------|--------|------|
| **步进** | "开始执行" | 执行一个任务 → 等确认 → 下一个 |
| **批量** | "全部执行" | 连续执行所有任务 → 最后报告 |
| **阶段** | "执行阶段1" | 执行一个阶段的所有任务 → 等确认 |
