# 内置 Harness 工程轨道工作流

本文件是 `doa-harness` skill 的自包含版本，供 WorkCopilot 在用户未安装独立 harness skill 时直接使用。

## 核心理念

Copilot 的上限取决于它运行在怎样的**工程轨道**上。
三层公式：**Context**（给什么）→ **Constraints**（守什么）→ **GC**（清什么）。

## 产出文件清单

| 文件 | 路径 | 用途 |
|------|------|------|
| copilot-instructions.md | `.github/copilot-instructions.md` | 项目级 Copilot 指令 |
| AGENTS.md | `.github/AGENTS.md` | Agent 角色约束与模块地图 |
| tasks.json | `.vscode/tasks.json` | 验证链自动化 |
| CODEOWNERS | 项目根目录 | 代码审阅权限 |
| rules/*.md | `rules/` 目录 | 编码规则文件 |

## 工作流（6 步）

### Step 1: 深度探测项目结构

递归扫描根目录及一级子目录，收集信号文件：

| 信号文件 | 技术栈 | 进一步读取 |
|----------|--------|-----------|
| `package.json` + `tsconfig.json` | Node/TypeScript | 读 dependencies 判断框架 |
| `*.csproj` / `*.sln` | .NET | 读 TargetFramework/Sdk |
| `pyproject.toml` / `requirements.txt` | Python | 读依赖判断框架 |
| `go.mod` | Go | 读依赖 |

**前端框架判断**（读 package.json dependencies）：
- `umi` → UmiJS | `next` → Next.js | `vue` → Vue | `react`(无umi/next) → React SPA | `@angular/core` → Angular

**同时检测**：
- 包管理器：`yarn.lock` / `pnpm-lock.yaml` / `package-lock.json`
- CSS 方案：`tailwind.config.*` / `.less` / `*.module.css`
- 状态管理：`zustand` / `dva` / `redux` / `pinia`
- 测试框架：`vitest` / `jest` / `xunit` / `pytest`
- CI/CD：`.github/workflows/` / `azure-pipelines.yml` / `.gitlab-ci.yml`

### Step 2: 交互确认

使用 `vscode_askQuestions` 收集：
1. **项目描述**：一句话概括
2. **确认探测结果**：展示技术栈，让用户确认或修正
3. **代码托管平台**：GitHub / GitLab / Azure DevOps
4. **团队 Owner 列表**：`@user1 @user2`（默认 `@todo-fill-team`）

### Step 3: 生成 5 类配置文件

#### 3.1 copilot-instructions.md

```
段落结构：
1. 项目概述 — 名称 + 各子项目入口
2. 技术栈 — 按子项目分段
3. 安全边界 — 按子项目分段
4. 前后端联动（仅 monorepo）
5. 验证命令 — 从 verify_chain 生成
6. 代码约定 — 按语言生成
7. 操作安全 — CI 平台 + 核心目录
8. 垃圾回收
```

头部加版本标记：`# Generated by doa-harness on {YYYY-MM-DD}`

#### 3.2 AGENTS.md

- 自动探测 `docs/`、`README.md` 等列为快速入口
- 模块地图：扫描目录并按规则标注权限

| 目录模式 | 标注 |
|----------|------|
| `core` `auth` `security` `infra` `migrations` | 🔒 需审批 |
| `shared` `common` `lib` `api` `contracts` | ⚠️ 注意复用 |
| `features` `modules` `pages` `components` `tests` | ✅ 可自由修改 |

总行数控制在 200 行以内。

#### 3.3 tasks.json

对每个子项目的 `verify_chain` 生成 task 列表：
- label 格式：`"{sub.name}:{step}"`
- 添加 `options.cwd`
- 总验证任务 `"verify"` dependsOn 所有子验证

包管理器感知：`yarn.lock` → yarn | `pnpm-lock.yaml` → pnpm | 默认 npm

#### 3.4 CODEOWNERS

GitHub → `CODEOWNERS` 根目录 | GitLab → 含 `[Section]` | Azure DevOps → 跳过

#### 3.5 rules/*.md

每条规则包含：规则名 + 原因 + 正确示例 ✅ + 错误示例 ❌

按技术栈生成：
- React/TS → `react-patterns.md`
- .NET → `dotnet-data-access.md`
- Python → `python-imports.md`
- 通用 → `no-secrets-in-code.md`

### Step 4: 注入 GC 指令

在 copilot-instructions.md 末尾追加垃圾回收段落。

### Step 5: 验证完整性

| 校验项 | 检查逻辑 |
|--------|---------|
| 文件存在性 | 5 类文件全部存在 |
| 验证链一致 | tasks.json ↔ copilot-instructions.md 验证命令匹配 |
| 模块地图覆盖 | AGENTS.md ↔ 实际目录不遗漏 |
| rules 引用 | AGENTS.md 引用 ↔ 实际 rules/*.md 文件 |

### Step 6: Day-1 验证指令

```
打开 Copilot Agent 模式，输入：
"读一下项目指令，然后告诉我：
 1. 哪些目录不能修改？
 2. 验证命令是什么？
 3. 哪些操作需要先确认？"
```

Copilot 能准确回答这 3 个问题 → Harness 生效。
